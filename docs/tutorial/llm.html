Package · com.aster.llm

llm 包：只保留流式模型调用。

当前项目没有非流式路径。`OkHttpStreamingChatClient` 负责把 `ChatRequest` 变成 HTTP 请求，再把每行 SSE 交给 `OpenAiCompatibleStreamParser`，统一输出 `ProviderStreamEvent`。DeepSeek V4 默认打开 thinking mode，并请求服务端在流式末尾返回真实 token usage。

四张图看懂 llm 包

架构图

OpenAiCompatibleProvider
  -> OpenAiCompatibleChatClient
       -> OkHttpStreamingChatClient
            -> OpenAiCompatibleStreamParser
                 -> ProviderStreamEvent
                      -> TextDelta
                      -> ReasoningDelta
                      -> ToolCallDeltaPart
                      -> UsageDelta
                      -> Done

ProviderStreamEvent 是供应商流和 AgentLoop 之间的统一边界。

文件结构图

llm/

  DeepSeekProvider.java                # DeepSeek 默认配置
  OkHttpStreamingChatClient.java       # HTTP SSE 客户端
  OpenAiCompatibleStreamParser.java    # raw JSON -> ProviderStreamEvent
  model/
    ChatRequest.java                   # 请求体
    Message.java                       # chat message 协议
    ProviderStreamEvent.java           # 统一流式事件
    TokenUsage.java                    # usage 和缓存 token

接入新 OpenAI-compatible 供应商时，优先新增 provider definition。

时序图

AgentLoop -> StreamingChatClient: stream(request)
Client -> LLM: POST /chat/completions
LLM -> Client: data: {...}
Client -> Parser: parse(data)
Parser -> Client: ProviderStreamEvent
Client -> AgentLoop: handler.onEvent(event)
AgentLoop -> EventBus: AgentEvent

Client 只读 SSE，Parser 负责理解兼容协议字段。

流程图

构造 ChatRequest
  -> 加入 stream=true
  -> DeepSeek 可带 thinking 字段
  -> POST SSE 请求
  -> 逐行读取 data
  -> 解析 content/reasoning/tool_call/usage
  -> 输出 ProviderStreamEvent
  -> [DONE] 结束

usage 是独立事件，不拼进 assistant 正文。

源码对照：DeepSeek 是默认，不是写死

OpenAiCompatibleProviderFactory.java 
public static OpenAiCompatibleProvider fromEnvWithDeepSeekDefaults() {
    return fromEnvWithDefaults(new DeepSeekProvider());
}

public static OpenAiCompatibleProvider fromEnvWithDefaults(OpenAiCompatibleProviderDefinition providerDefinition) {
    String providerName = envOrDefault(PROVIDER_ENV, providerDefinition.name());
    String baseUrl = envOrDefault(BASE_URL_ENV, providerDefinition.defaultBaseUrl());
    String model = envOrDefault(MODEL_ENV, providerDefinition.defaultModel());
中文解释

默认 provider 是 DeepSeek，但通过接口传进去。

通用环境变量可以覆盖 provider、baseUrl、model。

后面接 Kimi、Step，只要实现同一个 provider definition。

只有默认 DeepSeek provider 会打开 `thinkingEnabled`，通用 provider 覆盖时不会强行带 DeepSeek 私有能力。

源码对照：DeepSeek thinking 请求字段

ChatRequest.java 
return new ChatRequest(
        model,
        messages,
        tools,
        toolChoice,
        true,
        thinkingEnabled ? Map.of("type", "enabled") : null,
        thinkingEnabled ? reasoningEffort : null,
        Map.of("include_usage", true)
);
中文解释

`stream=true` 仍然表示流式输出。

`thinking.type=enabled` 打开 DeepSeek V4 thinking mode。

`reasoning_effort=high` 表示默认使用较强推理强度。

`stream_options.include_usage=true` 要求服务端在流式结束前补一包 usage。

这些字段只在 provider 支持时发送，不影响其他 OpenAI-compatible 服务。

源码对照：发送流式请求

OkHttpStreamingChatClient.java 
Request.Builder builder = new Request.Builder()
        .url(endpoint)
        .post(RequestBody.create(json, JSON))
        .header("Content-Type", "application/json")
        .header("Accept", "text/event-stream");

if (apiKey != null && !apiKey.isBlank()) {
    builder.header("Authorization", "Bearer " + apiKey);
}
中文解释

请求体是 OpenAI-compatible 的 JSON。

`Accept: text/event-stream` 告诉服务端：我要流式返回。

API Key 用 Bearer token 放进 Authorization 头。

源码对照：SSE 行转统一事件

OkHttpStreamingChatClient.java 
String data = line.substring("data:".length()).trim();
for (ProviderStreamEvent event : streamParser.parse(data)) {
    handler.onEvent(event);
    if (event instanceof ProviderStreamEvent.Done) {
        return;
    }
}
中文解释

每行 `data:` 后面是一个 JSON 增量。

OkHttp 客户端只负责读行，不把 OpenAI 原始结构泄漏给 AgentLoop。

`OpenAiCompatibleStreamParser` 把 raw JSON 转成 ProviderStreamEvent。

`[DONE]` 会转成 `ProviderStreamEvent.Done`，表示模型这轮输出结束。

源码对照：ProviderStreamEvent 到 AgentEvent

AgentLoop.java 
if (event instanceof ProviderStreamEvent.TextDelta delta) {
    publish(new AgentEvent.AssistantToken(delta.text()));
}
if (event instanceof ProviderStreamEvent.ReasoningDelta delta) {
    publish(new AgentEvent.ReasoningToken(delta.text()));
}
中文解释

ProviderStreamEvent 是 LLM 接入层的统一输出。

AgentEvent 是 Agent 主循环对 TUI、Web、日志暴露的事件协议。

DeepSeek 的 `reasoning_content` 先变成 `ReasoningDelta`，再变成 `ReasoningToken`。

两层事件分开后，新增供应商时只改 parser，不改 AgentLoop 的主逻辑。

TUI 才能把 thinking 和正文分开显示，Web SSE 也能直接转发同一批事件。

源码对照：每轮 token 用量来自 usage chunk

AgentLoop.java 
if (event instanceof ProviderStreamEvent.UsageDelta usage) {
    publish(new AgentEvent.UsageReported(
            usage.usage(),
            maxContextTokens
    ));
}
中文解释

usage 不是模型正文，所以不能拼到 assistant content 里。

`prompt_tokens` 是输入 token，`completion_tokens` 是输出 token。

DeepSeek 还会给 `prompt_cache_hit_tokens` 和 `prompt_cache_miss_tokens`，input 会优先按 cache + miss 计算。

AgentLoop 把它转成 `UsageReported` 事件，并附带最大上下文窗口。

EventBus 会给 usage 事件补 metadata，方便后续 Web 或审计日志按 run/sequence 关联。

TUI 不把 usage 插入历史区，而是更新输入框下面的预算行。
