Package · dev.agentmvp.tool

tool 包：所有工具先变成同一种语言。

Tool 层像一个转接板。本地 Java 工具、MCP 远端工具、Skill 加载工具，最终都对 AgentLoop 呈现成 `Tool`、`ToolCall`、`ToolResult`。

当前版本里，`ToolResult` 还会带成功失败和耗时，TUI 可以把工具调用渲染成可折叠的工具块。

四张图看懂 tool 包

架构图

AgentLoop
  -> ParallelToolExecutor
       -> ToolRegistry
            -> LocalToolExecutor
                 -> BuiltinTool
            -> McpToolExecutor
                 -> McpClient
       -> ToolResult
  -> ToolResultOffloader
  -> role=tool 消息

AgentLoop 只面对统一工具层，不区分本地工具还是 MCP 工具。

文件结构图

tool/
  ToolRegistry.java          # 工具注册和执行路由
  ParallelToolExecutor.java  # 多 tool_call 并行执行
  LocalToolExecutor.java     # 本地 handler 调度
  builtin/
    ReadTool.java            # 读取文件
    WriteTool.java           # 写文件
    BashTool.java            # 执行命令
    EditTool.java            # 精确替换
    LoadSkillTool.java       # 加载 Skill 全文
  result/
    ToolResultOffloader.java # 大结果写 JSONL

每个内置工具单独成类，最后统一注册到 ToolRegistry。

时序图

LLM -> AgentLoop: tool_calls[]
AgentLoop -> ParallelToolExecutor: executeAll()
ParallelToolExecutor -> ToolRegistry: execute(call)
ToolRegistry -> Local/MCP Executor: route
Executor -> ToolResult: result
AgentLoop -> Offloader: compact(result)
AgentLoop -> SessionStore: append(role=tool)

多个工具调用可以并行执行，但写回时仍按原顺序配对。

流程图

模型返回 tool_call
  -> 找到 tool name
  -> 按 ToolSource 路由
  -> 执行工具并记录耗时
  -> 大输出写入 JSONL
  -> 压缩文本写入 role=tool
  -> 发布 ToolCallDone 事件

源码对照：统一注册本地工具

ToolRegistry.java 
public void registerLocal(Tool tool, ToolHandler handler) {
    localExecutor.register(tool.name(), handler);
    register(tool);
}
中文解释

本地工具要登记两件事：模型看得见的工具定义，以及 Java 里的执行函数。

`registerLocal` 把两步合在一起，防止调用处只注册 schema、忘了注册 handler。

这就是“最后统一注册到 ToolRegistry”的落点。

源码对照：每个工具一个类

BuiltinTools.java 
public static List<BuiltinTool> defaultTools(Path workingDirectory) {
    return List.of(
            new ReadTool(workingDirectory),
            new WriteTool(workingDirectory),
            new BashTool(workingDirectory),
            new EditTool(workingDirectory)
    );
}
中文解释

`BuiltinTools` 不实现具体逻辑，只负责组装默认工具列表。

`ReadTool`、`WriteTool`、`BashTool`、`EditTool` 各自继承同一个抽象基类。

以后加工具时，新增类并放进列表，不需要维护一个巨大 switch。

源码对照：本地还是 MCP，由 ToolRegistry 路由

ToolRegistry.java 
if (tool.source() == ToolSource.MCP) {
    // MCP 工具不侵入 AgentLoop。这里只按 ToolSource 路由到远程执行器。
    return mcpExecutor.execute(tool, call);
}
// 本地工具和 MCP 工具返回同一种 ToolResult，后续统一转成 role=tool 消息。
return localExecutor.execute(tool, call);
中文解释

AgentLoop 不需要知道工具在哪儿执行。

`ToolSource.MCP` 走远端执行器，本地工具走本地执行器。

无论哪条路，最后都返回 `ToolResult`，再写回 `role=tool`。

源码对照：ToolResult 带执行元信息

ParallelToolExecutor.java 
private ToolResult executeOneWithTiming(ToolCall call) {
    long start = System.nanoTime();
    ToolResult result = toolRegistry.execute(call);
    long elapsedMillis = Math.max(0, (System.nanoTime() - start) / 1_000_000);
    return result.withExecutionMetadata(call.function().name(), elapsedMillis);
}
中文解释

具体工具只负责返回业务结果。

并行执行器统一记录耗时，并补上工具名。

TUI 收到 `ToolCallDone` 后，就能显示成功失败、输出和 Took 时间。

长输出默认折叠，用户用 `ctrl+o` 展开或折叠。

源码对照：大工具结果卸载成 JSONL

ToolResultOffloader.java 
public String compact(ToolResult result) throws IOException {
    String renderedText = result.renderText();
    if (renderedText.length() <= maxInlineChars || outputDirectory == null) {
        return renderedText;
    }

    StoredFile storedFile = store(result, renderedText);
    return renderCompactMessage(result, renderedText, storedFile);
}
中文解释

小结果直接内联进 `role=tool`。

大结果写入 `workspace/artifacts/tool-results/*.jsonl`。

tool 消息里只保留 `jsonlPath`、`recordId` 和预览，避免上下文被大输出撑爆。

这一步不改变 tool_call_id，所以不会破坏工具协议配对。
