# Context Summary Prompt

你要把较早的对话历史压缩成一段摘要。

这份摘要用于后续开发继续接上上下文。它必须全面覆盖技术细节、代码模式、架构决策、用户反馈和当前进展，同时删除无关闲聊，避免丢失后续工作所需信息。

## 摘要前自检

在输出最终摘要前，先在内部按时间顺序核对历史内容，但不要输出核对过程。核对时重点确认：

1. 用户的明确请求和真实意图。
2. 你处理请求的方式，包括读过什么、改过什么、验证过什么。
3. 关键决策、技术概念、代码模式和架构取舍。
4. 具体细节，例如文件名、类名、方法名、函数签名、配置项、命令、测试结果。
5. 遇到过的错误、原因、解决方式和剩余风险。
6. 用户给过的纠正和偏好，尤其是“不要这样做”“应该这样做”“先别做”“只改模板”等指令。
7. 最新一轮对话正在处理什么，是否已经完成。

## 摘要目标

- 保留用户明确提出的需求。
- 保留用户纠正过的偏好、边界和反馈，尤其是“不要这样做”“改成这样”的指令。
- 保留已经确认的技术决策和架构取舍。
- 保留工具调用得到的重要事实。
- 保留文件路径、类名、方法名、错误信息、命令输出和测试结果。
- 保留已经完成的改动、仍在进行的工作、未完成的待办和下一步需要接着做的事项。
- 删除寒暄、重复解释、无关闲聊。
- 删除临时过程、无关推测和已经失效的计划。

## 推荐结构

按下面结构输出，确实没有内容的部分可以省略：

1. 主要请求和意图：用户明确要求过什么，最新目标是什么。
2. 关键技术概念：涉及的架构、模块、工具协议、状态流转或重要约束。
3. 文件和代码段：读过、改过或需要继续关注的文件路径、类名、方法名、关键代码片段。
4. 错误及解决方法：遇到过什么错误，原因是什么，如何处理，用户是否给过反馈。
5. 问题解决：已经解决的问题、仍在排查的问题和验证结果。
6. 所有用户消息：列出对理解意图重要的用户消息，尤其是最新消息和纠正性反馈。
7. 待办任务：用户明确要求但尚未完成的任务。
8. 当前工作：压缩发生前最近正在处理什么，做到哪一步。
9. 可选下一步：只列和最新用户请求直接相关的下一步，不要发散；如果上一项任务已完成且没有后续要求，可以省略。

## 输出结构示例

1. 主要请求和意图：
用户希望……；后来明确纠正为……；最新要求是……。

2. 关键技术概念：
- ContextWindowCache：……
- SystemReminderInjectHook：……
- Tool 协议：……

3. 文件和代码段：
- `src/main/java/com/aster/.../Example.java`
  - 重要性：……
  - 已做改动：……
  - 关键方法：`methodName(...)`
- `src/main/resources/prompts/.../example.md`
  - 重要性：……
  - 已做改动：……

4. 错误及解决方法：
- 错误：……
  - 原因：……
  - 解决：……
  - 用户反馈：……

5. 问题解决：
已完成……；验证命令是……；仍需注意……。

6. 所有用户消息：
- 用户说：“……”
- 用户说：“……”

7. 待办任务：
- ……

8. 当前工作：
压缩前最近正在处理……，已完成到……，相关文件是……。

9. 可选下一步：
如果继续，应先……；这一步直接对应用户最近的要求：“……”。

## 工具协议安全

- 不要输出 assistant tool_calls。
- 不要输出 role=tool 消息。
- 不要保留任何 tool_call_id。
- 摘要是一段普通文本，后续会被包装成新的 user message。
- 不要把 `<system-reminder>`、长期记忆、Skill 索引或工具结果当成用户原始输入来复述。

## 输出要求

- 用中文输出。
- 按时间顺序概括重要事实。
- 对未完成事项、用户偏好、当前进展要明确写出。
- 不要编造没有在历史里出现的信息。
- 不要输出分析过程，不要包 Markdown 代码块。
