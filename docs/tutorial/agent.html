Package · dev.agentmvp.agent

agent 包：主循环是 Agent 的心跳。

把 LLM 想成远程同事，`AgentLoop` 就是项目经理：给它上下文，接收回复，发现它要工具就安排工具，拿到结果再让它继续。

四张图看懂 agent 包

架构图

AgentLoop

  -> StreamingChatClient
       -> ProviderStreamEvent
  -> AssistantMessageBuilder
  -> ParallelToolExecutor
  -> SessionStore

AgentLoop 是主流程，EventBus 是对外事件出口。

事件模型放在 model 子包，主包保留主循环和总线。

时序图

User -> AgentLoop: run(input)
AgentLoop -> SessionStore: append(user)
AgentLoop -> ContextBuilder: build(messages)
AgentLoop -> LLM: stream(request)
LLM -> AgentLoop: ProviderStreamEvent
AgentLoop -> ToolExecutor: executeAll(toolCalls)
AgentLoop -> SessionStore: append(assistant/tool)
AgentLoop -> EventBus: AgentEvent

每个关键动作都能转成事件，方便 TUI、Web 和审计复用。

流程图

收到用户输入
  -> 写入 session
  -> 发布 turn_start
  -> 构造上下文
  -> 发起流式 LLM
  -> 收集正文 / thinking / tool_call
  -> 有工具则执行并回灌
  -> 无工具则结束本轮

事件流不是附属日志，而是后续 Web、日志、审计的统一协议。

组件群聊
User帮我读文件并修改。
AgentLoop我先把完整 session 交给 ContextBuilder，构造本轮消息。
LLM我先返回 reasoning_content，再请求 read 和 edit 两个工具。
ParallelToolExecutor收到，多个 tool_call 可以并行跑，并记录每个工具耗时。
AgentLoop工具结果已写回 role=tool，继续问模型生成最终答案。

源码对照：一次循环做什么

AgentLoop.java 
ContextBuildResult context = contextBuilder.build(sessionStore.loadMessages());
List<Map<String, Object>> tools = toolRegistry.toLlmToolSchemas();

ChatRequest request = ChatRequest.streaming(
        model,
        context.messages(),
        tools,
        tools.isEmpty() ? null : "auto",
        thinkingEnabled,
        reasoningEffort
);
中文解释

先从 SessionStore 取完整历史，再让 ContextBuilder 决定本轮发多少。

工具定义从 ToolRegistry 导出成 OpenAI-compatible schema。

没有工具时不传 `tool_choice=auto`，避免某些兼容模型报错。

DeepSeek 默认打开 thinking mode，所以请求里会带 `thinking` 和 `reasoning_effort`。

源码对照：AgentLoop 发出完整事件流

AgentLoop.java 
publish(new AgentEvent.TurnStarted(roundNumber));

ContextBuildResult context = contextBuilder.build(sessionStore.loadMessages());
publish(new AgentEvent.ContextBuilt(
        context.compressed(),
        context.beforeTokens(),
        context.afterTokens(),
        context.maxContextTokens()
));

publish(new AgentEvent.LlmRequestStarted(
        roundNumber,
        model,
        context.messages().size(),
        tools.size()
));

private void publish(AgentEvent event) {
    eventBus.publish(event);
}
中文解释

AgentLoop 只发布“发生了什么”，不关心谁来消费。

LLM 层先把供应商原始流转换成 ProviderStreamEvent，AgentLoop 再转换成 AgentEvent。

run、turn、context、LLM request、message、tool、usage 都会变成 AgentEvent。

AgentEventBus 会把事件包装成 AgentEventEnvelope，补上 runId、sessionName、sequence、timestamp。

TUI、Web SSE、日志都实现 AgentEventHandler，复用同一条事件流。

这样新增出口时不用改 AgentLoop，只新增一个 handler 并注册到 EventBus。

源码对照：工具结果怎么回灌

AgentLoop.java 
ToolResult result = allowedResults.next();
BeforeToolResultAppendContext resultContext = hookRegistry.apply(
        AgentHookPoints.BEFORE_TOOL_RESULT_APPEND,
        new BeforeToolResultAppendContext(
                sessionName,
                runId,
                call,
                result,
                result.renderText()
        )
);

sessionStore.append(Message.tool(
        result.toolCallId(),
        resultContext.toolMessageText()
));
    publish(new AgentEvent.ToolCallDone(
            result.toolCallId(),
            call.function().name(),
            resultContext.toolMessageText(),
            !result.error(),
            result.elapsedMillis()
    ));
中文解释

多个工具调用先交给并行执行器，减少等待时间。

每个结果都必须写成 `role=tool`，并带原始 `tool_call_id`。

大结果卸载不再硬编码在 AgentLoop，而是挂到 `BEFORE_TOOL_RESULT_APPEND` Hook。

这一步是协议配对：assistant 说“我要工具”，tool 消息负责回答“工具结果在这里”。

同时发出 `ToolCallDone` 事件，让 TUI 能显示工具块、成功失败和耗时。
