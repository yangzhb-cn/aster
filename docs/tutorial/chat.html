Package · dev.agentmvp.chat

chat 包：消息字段放错位置，协议就会坏。

OpenAI-compatible Chat API 不是普通聊天记录。`assistant.tool_calls` 和 `tool.tool_call_id` 必须配对。DeepSeek thinking 还要求某些 assistant 消息保留 `reasoning_content`。

四张图看懂消息协议

架构图

Message
  role=system      -> 系统规则
  role=user        -> 用户输入 / 历史摘要
  role=assistant   -> content / reasoning_content / tool_calls
  role=tool        -> tool_call_id / content

ToolCall
  -> Function(name, arguments)

消息模型是 LLM 请求的协议边界，不是普通字符串列表。

文件结构图

llm/model/
  Message.java       # system/user/assistant/tool 消息
  ToolCall.java      # 完整工具调用
  ToolCallDelta.java # 流式工具调用片段
  ChatRequest.java   # messages + tools + thinking
core/agent/
  AssistantMessageBuilder.java # 拼接 assistant content/tool_calls
core/context/
  ToolProtocolValidator.java   # 校验工具协议配对

教程叫 chat 包，是为了单独讲消息协议；源码模型在 llm/model 下。

时序图

LLM -> AgentLoop: assistant(tool_calls)
AgentLoop -> SessionStore: append assistant
AgentLoop -> ToolRegistry: execute
ToolRegistry -> AgentLoop: ToolResult
AgentLoop -> SessionStore: append tool(tool_call_id)
ContextBuilder -> Validator: validate pair
AgentLoop -> LLM: messages with paired tool result

assistant 的 tool_calls 和 tool 的 tool_call_id 必须一一对应。

流程图

构造请求消息
  -> system 放规则
  -> user 放用户输入或摘要
  -> assistant 放模型回复
  -> assistant 可带 tool_calls
  -> tool 必须带 tool_call_id
  -> 压缩后重新创建摘要消息
  -> 校验后发给 LLM

压缩旧消息时不能保留残留 tool_calls。

四种角色

system系统规则，比如 Skill 索引、行为约束。
user用户输入，也用于干净的历史摘要。
assistant模型回复，可以带 tool_calls。
tool工具结果，必须带 tool_call_id。

源码对照：Message 模型

Message.java 
@JsonIgnoreProperties(ignoreUnknown = true)
@JsonInclude(JsonInclude.Include.NON_EMPTY)
public record Message(
        String role,
        String content,
        @JsonProperty("reasoning_content") String reasoningContent,
        @JsonProperty("tool_calls") List<ToolCall> toolCalls,
        @JsonProperty("tool_call_id") String toolCallId
) {
    public Message {
        // 空数组对很多 chat API 没有意义，这里统一归一化，避免序列化出多余字段。
        if (toolCalls == null) {
            toolCalls = List.of();
        }
    }
中文解释

`role` 决定这条消息是谁说的。

`reasoning_content` 只应该出现在 assistant 消息上，用来保存 DeepSeek thinking 内容。

`tool_calls` 只应该出现在 assistant 消息上。

`tool_call_id` 只应该出现在 tool 消息上，用来配对上一条 assistant 工具调用。

空字段不序列化，减少兼容模型误判。

源码对照：用工厂方法防止字段乱放

Message.java 
public static Message assistantToolCalls(String content, String reasoningContent, List<ToolCall> toolCalls) {
    // DeepSeek thinking + tool call 场景下，reasoning_content 后续请求还要带回 API。
    return new Message("assistant", content, reasoningContent, toolCalls, null);
}

public static Message tool(String toolCallId, String content) {
    return new Message("tool", content, null, List.of(), toolCallId);
}
中文解释

assistant 工具调用消息可以带 `tool_calls` 和 `reasoning_content`，但不能带 `tool_call_id`。

tool 结果消息只带 `tool_call_id`，不带 `tool_calls`，也不带 `reasoning_content`。

这种小工厂方法能减少调用方手写字段导致的协议错误。
