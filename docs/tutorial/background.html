Package · com.aster.app.background

background 包：先把后台任务框架搭起来。

后台任务不应该挤进当前对话主线程。当前实现提供统一入口、JSONL 持久化、调度器、执行器、事件总线和通知出口，并已经接入 `memory_extract`：每轮主对话完成后，通过事件监听触发后台长期记忆抽取。

四张图看懂 background 包

架构图

AgentRuntime
  -> BackgroundTaskManager
       -> JsonlBackgroundTaskStore
       -> BackgroundTaskScheduler
            -> BackgroundTaskExecutor
                 -> BackgroundTaskHandler
                 -> BackgroundTaskEventBus
                      -> BackgroundTaskNotificationHandler
                           -> NotificationSink
                                -> TUI 状态栏

Manager 是外部入口，Scheduler 管时间，Executor 管执行和记录，EventBus 管通知出口。

文件结构图

app/background/
  BackgroundTaskManager.java        # 后台任务统一入口
  BackgroundTaskScheduler.java      # immediate/delay/interval 调度
  BackgroundTaskExecutor.java       # 执行 handler，写 TaskRun
  BackgroundTaskStore.java          # 持久化接口
  JsonlBackgroundTaskStore.java     # tasks.jsonl / runs.jsonl
  BackgroundTaskEventBus.java       # 后台事件分发
  BackgroundTaskNotificationHandler.java # 事件转通知
  NoopTaskHandler.java              # 教学版占位任务
memory/
  MemoryExtractionTaskHandler.java  # 长期记忆抽取任务
  MemoryExtractionHook.java         # afterRun Hook，提交 memory_extract 后台任务
  model/                            # Task、Trigger、Action、Run

模型类放在 `model` 子包，主包只保留执行框架。

时序图

用户/工具 -> Manager: create(task)
Manager -> Store: saveTask(task)
Manager -> Scheduler: schedule(task)
Scheduler -> Executor: execute(task)
Executor -> Handler: handle(task)
Executor -> Store: appendRun(run)
Executor -> EventBus: TaskCompleted/TaskFailed
EventBus -> NotificationSink: 更新外部出口

任务完成时不打断当前对话，只把结果写入 runs.jsonl 并通知状态栏。

流程图

启动 runtime
  -> 读取 workspace/tasks/tasks.jsonl
  -> 恢复 enabled 任务
  -> 到点触发
  -> 找到支持 action 的 handler
  -> 写入 workspace/tasks/runs.jsonl
  -> 发布后台任务事件
  -> TUI 底部显示完成/失败计数

这个流程给长期记忆、定时通知、异步索引等任务复用。

源码对照：后台任务统一入口

BackgroundTaskManager.java 
public BackgroundTask create(BackgroundTask task) throws IOException {
    store.saveTask(task);
    scheduler.schedule(task);
    return task;
}
中文解释

外部只找 Manager，不直接操作 store 或 scheduler。

先保存任务定义，再注册调度，重启后还能从 JSONL 恢复。

以后工具层想创建后台任务，也应该调用这个入口。

源码对照：调度器只关心什么时候跑

BackgroundTaskScheduler.java 
ScheduledFuture<?> future = switch (type) {
    case "immediate" -> scheduler.schedule(() -> executor.execute(task), 0, TimeUnit.SECONDS);
    case "delay" -> scheduler.schedule(() -> executor.execute(task), delaySeconds, TimeUnit.SECONDS);
    case "interval" -> scheduleInterval(task);
    default -> throw new IllegalArgumentException("Unsupported background task trigger: " + type);
};
中文解释

教学版先用 `ScheduledExecutorService`，足够表达立即、延迟和间隔任务。

Scheduler 不知道任务内容是什么，只负责在合适的时间调用 Executor。

以后要支持 cron，可以替换或增强这一层。

源码对照：执行器写记录并发事件

BackgroundTaskExecutor.java 
BackgroundTaskHandler handler = findHandler(task);
String message = handler.handle(task);
TaskRun run = TaskRun.success(task, startedAt, Instant.now().toString(), message);
store.appendRun(run);
eventBus.publish(new BackgroundTaskEvent.TaskCompleted(run));
中文解释

Executor 不写具体业务逻辑，只按 action 找对应 handler。

每次运行都会写 `workspace/tasks/runs.jsonl`，方便审计和回溯。

完成或失败都会发布事件，外部可以接 TUI、Web、日志。

源码对照：策略 B，只更新底部状态栏

TuiNotificationSink.java 
@Override
public void backgroundTaskCompleted(TaskRun run) {
    window.showBackgroundTaskCompleted(run.taskName());
}
中文解释

后台任务完成时不插入对话历史，避免打断当前主对话。

TUI 只在底部状态栏显示完成/失败计数。

真正的任务记录已经写入 JSONL；用户想看细节时，再问 Agent 或后续加命令查看。
