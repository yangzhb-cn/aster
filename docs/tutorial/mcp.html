Package · dev.agentmvp.mcp

mcp 包：把外部工具翻译成项目内部 Tool。

MCP 的协议细节被包在 `McpClient`、`McpToolAdapter`、`McpToolExecutor` 里。AgentLoop 只看见普通 `Tool`，不需要知道背后是本地 stdio 进程还是 HTTP JSON-RPC。

四张图看懂 mcp 包

架构图

workspace/mcp.json
  -> McpConfigLoader
  -> McpClientFactory
       -> StdioMcpTransport / HttpMcpTransport
  -> McpClient
       -> initialize
       -> tools/list
  -> McpToolAdapter
  -> ToolRegistry
  -> McpToolExecutor(tools/call)

MCP 是外部协议，进入项目后统一变成内部 Tool。

文件结构图

mcp/
  McpClient.java             # JSON-RPC 客户端
  McpToolLoader.java         # initialize + tools/list + register
  McpToolAdapter.java        # MCP tool -> Tool
  McpToolExecutor.java       # tools/call 执行
  config/                    # mcp.json 加载和客户端工厂
  transport/                 # stdio / HTTP 传输
  server/                    # 教学版本地 MCP Server
  model/                     # JsonRpcRequest/Response/Error

client、transport、server、model 分开，方便只接入本地或远程 MCP。

时序图

Factory -> McpConfigLoader: loadIfExists()
Factory -> McpClientFactory: create(serverConfig)
McpClient -> MCP Server: initialize
McpClient -> MCP Server: tools/list
McpToolLoader -> ToolRegistry: register(tool)
LLM -> ToolRegistry: call MCP tool
McpToolExecutor -> MCP Server: tools/call

安装具体 MCP 时，只需要补本地 `workspace/mcp.json`。

流程图

读取 mcp.json
  -> 没有配置：跳过
  -> 有配置：创建 transport
  -> initialize
  -> tools/list
  -> 适配成内部 Tool
  -> 注册 ToolRegistry
  -> LLM 调用时发 tools/call

AgentLoop 不碰 JSON-RPC，因此不会被 MCP 细节污染。

源码对照：通过 workspace/mcp.json 接入外部 MCP

AgentRuntimeFactory.java 
McpConfig mcpConfig = new McpConfigLoader(objectMapper).loadIfExists(WorkspacePaths.MCP_CONFIG);
McpClientFactory clientFactory = new McpClientFactory(httpClient, objectMapper);
McpToolLoader toolLoader = new McpToolLoader(toolRegistry, mcpToolExecutor);

for (McpServerConfig serverConfig : mcpConfig.servers()) {
    // 每个 MCP Server 都会 initialize，再 tools/list，最后注册进 ToolRegistry。
    toolLoader.load(clientFactory.create(serverConfig));
}
中文解释

没有 `workspace/mcp.json` 时返回空配置，Agent 仍然只用内置工具启动。

有 `workspace/mcp.json` 时，配置里的每个 server 都会变成一个 `McpClient`。

本地 MCP 用 stdio 启动子进程；HTTP MCP 走 POST JSON-RPC。

配置形状：先不安装具体 MCP，也能保留接入能力

workspace/mcp.json.example

  "mcpServers": {
  }
}
中文解释

仓库提交空模板，表示当前不绑定任何具体 MCP。

真实 `workspace/mcp.json` 是本地配置，已加入 `.gitignore`。

initialize
tools/list

源码对照：客户端发现工具

McpClient.java 
public List<Tool> listTools() throws IOException {
    // tools/list 是 MCP 的工具发现入口。
    // 注意这里直接转成项目自己的 Tool，而不是暴露 McpTool 给 AgentLoop。
    JsonRpcResponse response = send("tools/list", Map.of());
    List<Tool> tools = new ArrayList<>();

    for (var toolNode : response.result().path("tools")) {
        tools.add(McpToolAdapter.toTool(serverId, toolNode, objectMapper));
    }

    return tools;
}
中文解释

MCP 返回的是自己的 tools/list 结构。

项目不会把 MCP 结构直接散到 AgentLoop，而是马上适配成内部 `Tool`。

这样 MCP 是插件式能力，不会污染主循环。

源码对照：本地 MCP Server 处理方法

LocalMcpServer.java 
return switch (request.method()) {
    case "initialize" -> handleInitialize(request);
    case "tools/list" -> handleToolsList(request);
    case "tools/call" -> handleToolsCall(request);
    default -> error(request.id(), JsonRpcProtocol.METHOD_NOT_FOUND, "Method not found: " + request.method());
};
中文解释

教学版本地 MCP Server 只支持三个方法。

`initialize` 宣告能力，`tools/list` 暴露工具，`tools/call` 执行工具。

未知方法返回 JSON-RPC 标准错误。
