Package · dev.agentmvp.context

context 包：压缩上下文，但不切断协议。

这里的核心不是“少发点字”，而是安全地少发。`ContextBuilder` 按 user turn 切块，避免把 assistant 的工具调用和 tool 结果拆开。

为什么不直接按 token 硬切？

system系统指令，通常保留
user turn用户输入 + 后续 assistant/tool
orphan恢复历史时出现的半截消息
summary旧 turn 被摘要成干净 user 消息

源码对照：先判断是否压缩

ContextBuilder.java 
int beforeTokens = tokenEstimator.estimate(sessionMessages);
int threshold = (int) (options.maxContextTokens() * options.compressThreshold());

if (beforeTokens < threshold) {
    // 即使不压缩，也要校验协议。因为历史可能来自恢复的 session，
    // 里面可能已经存在孤立工具结果或残留 tool_calls。
    ToolProtocolValidator.validate(sessionMessages);
    return new ContextBuildResult(List.copyOf(sessionMessages), false, beforeTokens, beforeTokens, null);
}
中文解释

先估算当前消息会占多少 token。

没到阈值也要校验工具协议，防止恢复出来的历史本来就是坏的。

如果没压缩，直接返回一份不可变副本，SessionStore 仍保留原始历史。

源码对照：摘要必须重建成干净消息

ContextBuilder.java 
if (!summaryInput.isEmpty()) {
    summary = summarizer.summarize(summaryInput);

    // 摘要必须重建成一条干净的普通消息，不能克隆旧 assistant 消息。
    // 否则旧 tool_calls 可能残留，而对应工具结果已经被摘要吃掉。
    finalMessages.add(Message.user(
            "以下是系统自动生成的历史对话摘要，不是用户的新请求：\n\n" + summary
    ));
}
中文解释

旧消息被摘要后，不再保留原来的 assistant/tool 结构。

新的摘要消息用 `Message.user` 创建，所以不会带残留 `tool_calls` 字段。

这就是防止“assistant 有 tool_call，但 tool 结果被压缩掉”的关键。
