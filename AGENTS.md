# AGENTS.md

本文件只记录用户偏好和协作规则。项目结构、架构边界、核心流程、命令和模块说明，优先查看 `docs/ai-readme/README.md`。

## 协作偏好

- 默认用中文沟通，表达直接、简洁、具体。
- 教学模式下，先讲清楚“改动前是什么、改动后是什么、涉及哪些文件、为什么这么做”，再动代码。
- 需求存在多种解释时，不要静默选择；先说明取舍，必要时提问。
- 用户明确说“改动吧 / 开始吧 / 可以”后，直接实现、验证、交付，不停在方案层。
- 发现用户的判断有风险时，可以指出，但要给出可执行替代方案。

## 实现偏好

- 优先复用当前项目已有逻辑和分层，不要为了单次需求新增大抽象。
- 改动要小，文件要少；不要顺手重构、格式化或清理无关代码。
- 新能力优先走 extension、hook、event、tool registry、runtime 装配等现有扩展点。
- 不要轻易大改 AgentLoop、Context、Session、Tool 协议等核心链路；必须改时要说明原因和影响面。
- 新增注释用中文；新增类写类注释；核心方法写方法注释。
- 不要回滚用户已有改动；不要提交、推送，除非用户明确要求。

## 文档规则

- `AGENTS.md` 只放偏好和协作规则，不堆项目架构细节。
- 涉及架构、跨模块修改或不熟悉目录时，先读 `docs/ai-readme/generated/`。
- 涉及业务规则、历史坑点、用户偏好沉淀时，读并更新 `docs/ai-readme/manual/`。
- 涉及代码改动、架构变化、功能新增、入口能力变化或经验沉淀时，交付前评估是否同步 `docs/ai-readme/`。

## 验证规则

- 代码改动后跑相关测试；核心链路、Session、Tool、Plan/Team、后台任务、schedule、Web/IM/TUI 事件映射改动后至少跑 `mvn test`。
- Web 前端 JS 改动后跑 `node --check src/main/resources/web/assets/app.js`。
- 交付前跑 `git diff --check`。
- 纯文档修改可以不跑测试，但最终说明要写清楚。
